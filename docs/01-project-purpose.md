# 01 — プロジェクトの目的

## 一文での定義

**ユーザーがInterstitial広告を見なくて済むようにする、デスクトップブラウザ拡張機能を作る。**

---

## 達成すること（In Scope）

1. **Interstitial広告を自動検出する**
   - ページ閲覧中に表示されるInterstitialを、視覚的特徴（配置、サイズ、z-index等）から自動的に見つけ出す。
2. **検出したInterstitialを消す**
   - 優先：閉じるボタンを自動クリック
   - フォールバック：CSSで強制的に非表示化
3. **正規モーダルを誤爆しない**
   - ログイン、クッキー同意、画像ライトボックス等は触らない
4. **ユーザーが調整できる**
   - サイト単位のホワイトリスト
   - 拡張機能のオン/オフ
5. **Chrome / Firefox の両方で動く**
   - Manifest V3 ベース

---

## 達成しないこと（Out of Scope）

このプロジェクトは**広告ブロッカーではない**。以下は明確に**やらない**。

1. **バナー広告・サイドバー広告のブロックはしない**
   - 対象は Interstitial（ページを覆うタイプの広告）のみ
2. **広告ネットワークごとのDOMパターン辞書を作らない**
   - 過去プロジェクトで限界に突き当たったため
3. **ホスト名ベースのブロックリスト（EasyList等）は使わない**
   - 広告ブロッカーに近づきすぎる
4. **VPN / Proxy / DNS / MITM 等のネットワーク介入はしない**
   - スコープ外。拡張機能だけで完結させる
5. **広告の表示自体を完全に防ぐことは目指さない**
   - 「Interstitialが表示されても、ユーザーを止めない」ことを重視する
   - 表示はされるが、すぐ閉じる / 無視できる、という状態でOK

---

## プロジェクトの動機と前提

### 動機

Interstitial広告はユーザーの閲覧体験を著しく阻害する。コンテンツを読む、リンクを辿る、といった自然な操作の途中に割り込み、閉じるボタンも遅延表示で見つけにくい。

### 過去の試行

以前のプロジェクトでは、**ブラウザのデバッグポートに接続してDOMを監視し、パターンマッチングでコードを取り除く**方式を実装していた。しかし：

- 広告ネットワークごとにDOM構造が異なる
- 同じ広告ネットワークでもDOM構造が頻繁に変わる
- 「どの要素を消せばInterstitialが消えるか」の判断が実際のページでは難しかった

この経験から、**DOM構造のパターンではなく、視覚的表現の特徴**で判定する方針に転換する。

### 前提

- デスクトップブラウザ拡張（Chrome / Firefox）
- Interstitialは「幅広いサイト全般」を対象。特定サービス（YouTube等）限定ではない
- 「閉じるボタンをクリックして広告インプレッションを発生させる」ことは許容（動けば良い）

---

## 成功の定義（MVP段階）

以下が満たされれば、MVPとして成功とする：

1. Google AdSense の Interstitial が自動で閉じられる
2. 一般的なニュースサイト等に出現する「コンテンツを覆うモーダル型広告」が自動で閉じられる
3. ログインモーダルやクッキー同意バナーを誤爆しない
4. ユーザーが特定サイトで拡張を無効化できる
5. Chrome で動作する（Firefox は MVP 後）

---

## 非機能要件

- **軽量**：ページロードを遅延させない（100ms以内のオーバーヘッド）
- **低誤検出**：100サイト程度の手動テストで誤爆1件以下が目安
- **デバッグ可能**：検出ロジックの判断根拠がログに残る
- **プライバシー**：外部サーバーへの通信なし（ホスト側の統計収集もしない）
