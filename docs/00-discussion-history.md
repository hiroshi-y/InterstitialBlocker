# 00 — ディスカッション経緯（参考資料）

Claude（web版）と安川氏が行った設計ディスカッションの経緯を記録。
実装中に方針がブレたとき、元の議論に立ち返るための資料。

---

## ディスカッションの出発点

### ユーザーからの初期提起

> Interstitial Adsはとても面倒なので、元々表示されないようにするか、または表示されたらすぐクリックするか、というアプローチで、ユーザーがInterstitial Adsを見なくて済むようにしたい。
> 
> 以前のプロジェクトでは、ブラウザのデバッグポートに接続してDOMを監視するアプローチで、該当のコードが挿入されたらパターンマッチングでそのコードを取り除く、というものを動かしていたが、実際にInterstitialに出会っても、DOMのどの部分をフィルターすれば消えるのかの判断が難しかった。
> 
> 別のアプローチを検討するため、ディスカッションしたい。
> - VPNサーバーを使う方法、プロキシを使う方法もあるが、これらはコードをフィルタして除外するアプローチなので、前回のデバッグポート方式と似たような結果になることが考えられる
> - Interstitialが表示されているとき、クリックイベントを実際に拾えるのは1つのオブジェクトだけになっているはず。これを検出し、そのオブジェクトにクリックイベントを送信できれば、いったん表示されたInterstitialを消せるのではないか
> - このプロジェクトでは、広告ブロッカーを作るわけではない。Interstitialだけを無効化したい

### 決定事項（ディスカッションで確定）

- 対象プラットフォーム：**デスクトップブラウザ拡張（Chrome/Firefox）**
- 対象サイト：**幅広く全般**
- 閉じるボタン自動クリック（広告インプレッション発生）：**OK、動けば良い**

---

## アプローチ検討の核心

### 過去プロジェクトの学びと、その乗り越え方

過去の「DOMパターンマッチング」は、広告ネットワーク差異とDOM構造変動のせいで機能しなかった。
この教訓から、**DOM構造ではなく視覚的表現の特徴**で判定するという方針を採用。

理由：
- Interstitialは必然的に画面を覆う必要がある → CSSの選択肢は限定的
- `position: fixed`, 高 `z-index`, 大きなサイズ、半透明背景 ... これらは広告ネットワーク間で共通
- DOM構造のような頻繁な変更が起きにくい

### ユーザーの「クリック検出」アイデアについて

ユーザーの当初提案は「クリックイベントを拾える要素は1つだけ → それを見つけてクリック」だった。

実際に詰めていくと：
- Interstitial表示中でも「閉じるボタン」「広告本体」「オーバーレイ背景」等、複数のクリック対象が存在する
- 広告本体をクリックすると広告主サイトに遷移してしまう（避けたい）
- 「閉じるボタン」だけを正確に識別する必要がある

→ このアイデアは**レイヤー2の閉じるボタン探索**として部分採用。
単独ではなく、視覚特徴検出とセットで使う。

### 視覚特徴スコアリング：採用

ポジティブ要素：
- `position: fixed/absolute`
- 高 `z-index`
- ビューポート占有率（50%以上、80%以上で加点）
- 半透明オーバーレイ背景
- iframe の存在
- id/class 名のヒント（`interstitial`, `overlay` 等）
- body のスクロールロック状態

ネガティブ要素（誤爆防止）：
- `input[type="password"]` → ログイン
- フォーム要素
- cookie/consent/gdpr テキスト
- 年齢確認・利用規約テキスト
- 単一の大きな画像 → ライトボックス

### 閉じるボタンをクリックする vs 消す

議論の結果、**両方使う**（多層防御）。

優先：閉じるボタン自動クリック
- ユーザー合意済み（広告インプレッション発生OK）
- 広告スクリプトの正規フローに乗るため、後処理（クリーンアップ）が自然に走る
- サイト運営者の収益に配慮

フォールバック：強制除去
- 閉じるボタンが見つからない場合
- 閉じるボタンをクリックしても効かない場合
- 最大待機時間タイムアウト時

### 予防的CSSの発想

「Interstitialを消せなくても、ユーザーを止めなければ体感は大きく改善する」という発想から、
`overflow: auto !important` などのCSSを `document_start` で注入して、広告がスクロールロックしても効かないようにする。

広告を消さなくても、ユーザーは無視できる。
これはレイヤー1として採用。

### 不採用になったアプローチ

| アプローチ | 不採用理由 |
|-----------|-----------|
| VPN / Proxy | HTTPS で MITM 必要、UX悪化、広告ブロッカー化 |
| DNSブロック | パターン辞書問題、Interstitialだけを狙えない |
| デバッグポート + DOMパターン | 過去プロジェクトで失敗済み、メンテ不能 |
| EasyList等のブロックリスト流用 | 広告ブロッカー化、スコープ逸脱 |

## 多層防御アーキテクチャの論理

単一手法に依存しない理由：

1. **Layer 1（予防的CSS）**
   - 消せない場合のフォールバック的な「諦め動作」を提供
   - コストゼロ、副作用最小
   
2. **Layer 2（視覚検出 + クリック）**
   - メインの解決策
   - 理想的なケースで動作
   
3. **Layer 3（強制除去）**
   - Layer 2 が効かないケースのフォールバック
   - `display: none` + style guard
   
4. **Layer 4（遅延表示対応）**
   - Google AdSense 等の「閉じるボタンが数秒遅延」に対応
   - Layer 2 の時間軸拡張

これらを組み合わせることで、単一手法では取りこぼすケースをカバーする。

## 重要な設計判断の記録

### 判断1：`display: none` vs `element.remove()`

採用：`display: none`

理由：
- ページのJavaScriptが要素を参照している可能性（remove で参照エラー）
- 広告ネットワークのトラッキングが「読み込まれなかった」と誤認し、リトライされる可能性
- 表示さえ消えれば目的達成

### 判断2：`element.click()` vs フルマウスイベント

採用：フルマウスイベント（pointerdown, mousedown, pointerup, mouseup, click）

理由：
- 広告スクリプトによっては `click` だけでは反応しないケースがある
- 正規ユーザーのクリックを模倣することで互換性を最大化

### 判断3：スコア閾値をハードコードしない

採用：設定値で可変、デフォルト7

理由：
- サイトによって適切な閾値は違う
- ユーザーが調整できる方が実用的
- 上級者向け機能としてpopupから変更可能にする

### 判断4：外部サーバーへの通信はしない

採用：完全ローカル動作

理由：
- プライバシー優先
- テレメトリなし
- 設定は `chrome.storage.sync` のみ（Googleアカウント内）

### 判断5：ユーザークリック起因の要素は触らない

採用：直近2秒以内のクリック後に出現した要素はスキップ

理由：
- 画像ライトボックス、ドロップダウンメニュー等の誤爆防止
- ユーザーの明示的な意思で開いたUIを尊重

## やり取りの中で出てきた興味深い観察

- Interstitialの「視覚的特徴」は広告ネットワーク間で意外と共通している
- 閉じるボタンは遅延表示されることが多い（Googleのポリシーで変わってきている面もある）
- 閉じるボタンのアイコンは多様（×、✕、SVG、画像、Unicode等）→ 複数の検出戦略が必要
- cross-origin iframe はcontent scriptで `all_frames: true` にすれば中身を操作できる

---

## 実装時に立ち返るべき原則

ディスカッションを通じて確立された原則：

1. **広告ブロッカーを作らない**。Interstitial限定。
2. **DOMパターン辞書を作らない**。視覚特徴で判定。
3. **誤検出を避ける**。見逃しより誤爆が問題。
4. **ユーザーのUI操作を尊重する**。ユーザー起因のモーダルには触らない。
5. **動けば良い**。閉じるボタンクリックで広告インプレッションが発生するのはOK。
6. **プライバシー最優先**。外部通信なし。

これらの原則と実装の整合性に迷ったとき、このドキュメントに立ち返ること。
