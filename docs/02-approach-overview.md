# 02 — アプローチ概要（検討過程と選定理由）

設計ディスカッションで検討した複数のアプローチと、採用/不採用の理由を記録しておく。
Claude Codeが実装中に「なぜこの方針なのか」を見失わないための参照資料。

---

## 検討したアプローチ一覧

### ❌ A. DOMパターンマッチング（旧プロジェクトの方式）

**方法**：ブラウザデバッグポートに接続、DOM変更を監視し、広告ネットワークごとの既知セレクタや構造パターンに一致する要素を除去。

**不採用理由**：
- 広告ネットワークごとにDOM構造が違う
- 同じネットワークでもDOM構造が頻繁に変わる
- メンテナンスコストが青天井
- 実際のページで「どの要素を消せば良いか」の判断が困難だった（実績あり）

### ❌ B. VPN / Proxy / DNSブロック

**方法**：広告配信ドメインへのリクエストをネットワークレベルでブロック。

**不採用理由**：
- HTTPS が主流 → MITM proxy が必要 → 証明書インストールでUX悪化
- Aと同じ「パターン辞書」の問題を抱える（ブロックリスト管理）
- 「広告ブロッカー」そのものになってしまい、プロジェクトスコープと不一致
- Interstitialだけを狙い撃ちできない

### ❌ C. クリック検出のみ（ユーザーの当初提案）

**方法**：Interstitial表示中は「クリックイベントを拾える要素が1つだけ」になることを利用し、その要素を検出してクリックを送る。

**部分採用**：アイデアは有用だが、単独では不十分。
- 実際には「閉じるボタン」「広告本体（クリックで広告主サイトへ遷移）」「オーバーレイ背景」など複数のクリック対象が存在する
- 「閉じるボタン」だけを正確に識別する必要がある
- → 視覚特徴検出と組み合わせて使う（レイヤー2で採用）

### ✅ D. 視覚的特徴によるスコアリング検出（採用）

**方法**：CSS特性（position, z-index, サイズ、半透明背景等）で「Interstitialらしさ」をスコアリング。

**採用理由**：
- 視覚的表現は広告ネットワーク間で共通しやすい（全画面を覆う必要がある以上、CSSの選択肢は限られる）
- DOM構造のような頻繁な変更が起きにくい
- 広告ネットワーク非依存で動作する
- 1つのアルゴリズムで幅広く対応できる

### ✅ E. 閉じるボタンの自動クリック（採用）

**方法**：検出したInterstitial内で閉じるボタン候補を探し、自動クリック。

**採用理由**：
- ユーザー合意：「閉じるボタンをクリックして広告インプレッションを発生させることはOK（動けば良い）」
- サイト運営者の収益を不当に損なわないメリットもある
- ブラウザの正規イベントフロー内で動作するため、広告スクリプトが自然に後処理（クリーンアップ）を行う

### ✅ F. 強制除去（フォールバック）（採用）

**方法**：閉じるボタンが見つからない/効かない場合、CSSで強制非表示化。

**採用理由**：
- 閉じるボタンが遅延表示する、または存在しないケースへの保険
- 「ユーザーがInterstitialを見ずに済む」という目的達成のため必要

### ✅ G. 予防的CSS注入（採用）

**方法**：ページロード時に、body scroll lock を無効化するCSSを user stylesheet として注入。

**採用理由**：
- Interstitialを消せなくても「ユーザーを止めない」状態にできる
- 体感的な改善が大きい
- 副作用が小さい（正規モーダルへの影響も最小限）

## 採用した全体戦略：多層防御

単一の方法に頼らず、4つのレイヤーで冗長化する：

```
┌─────────────────────────────────────────────────┐
│ レイヤー1: 予防的CSS注入                         │
│   document_start で scroll lock 解除            │
│   → 消せなくても「止まらない」状態を作る         │
└─────────────────────────────────────────────────┘
                    ↓
┌─────────────────────────────────────────────────┐
│ レイヤー2: 視覚特徴検出 + 閉じるボタンクリック  │
│   MutationObserver でDOM変化を監視              │
│   スコアリングでInterstitial判定                │
│   閉じるボタン探索 → forceClick                 │
└─────────────────────────────────────────────────┘
                    ↓ （閉じるボタン見つからない/効かない）
┌─────────────────────────────────────────────────┐
│ レイヤー3: 強制除去                              │
│   display:none !important + style監視ガード     │
└─────────────────────────────────────────────────┘
                    ↓ （初回検出時に閉じるボタンが未出現）
┌─────────────────────────────────────────────────┐
│ レイヤー4: 遅延表示対策                          │
│   検出要素を継続監視、閉じるボタン出現を待つ    │
│   最大8秒でタイムアウト → レイヤー3へ           │
└─────────────────────────────────────────────────┘
```

詳しくは `03-architecture.md` と `04-detailed-design.md` を参照。

## 重要な設計判断

### 判断1：DOM構造ではなく視覚特徴で検出する

過去プロジェクトの反省。DOM構造に依存しない判定を第一原則とする。

### 判断2：削除ではなく非表示

`element.remove()` ではなく `display: none` を使う。
- ページのJavaScriptが要素を参照しているとエラーが出る可能性
- 広告ネットワークのトラッキングが誤動作する可能性

### 判断3：スコアリング閾値は可変に

サイトによって適切な閾値は違う。ハードコーディングせず、ユーザーが調整可能にする（popup UI経由）。

### 判断4：誤検出時のフィードバック機構を持つ

完璧な判定は不可能。ユーザーが「これは誤爆」「これは見逃し」をフィードバックできる仕組みを用意する。

### 判断5：all_frames: true で iframe 内にも注入

多くのInterstitialは iframe 内（Google AdSense等）。cross-origin iframe にも content script を注入する。
